# Phase 6: 程式碼審查

對照規格書，審查實作程式碼的完整性和品質。

## 前置條件
- 功能規格書已完成
- 實作程式碼已完成
- 測試已撰寫並通過

## 審查流程

### 1. 規格符合度審查

逐一對照規格書中的驗收條件：

```
| 驗收條件 | 狀態 | 對應程式碼 | 備註 |
|---------|------|-----------|------|
| AC-1: ... | ✅/❌ | file.ts:L42 | |
| AC-2: ... | ✅/❌ | file.ts:L78 | |
```

### 2. 程式碼品質審查

- **可讀性**：命名清晰、結構合理、適當的註解
- **可維護性**：低耦合、高內聚、遵循 SOLID 原則
- **一致性**：與專案既有風格一致
- **複雜度**：函式是否過長？邏輯是否過於複雜？

### 3. 安全性審查

對照安全準則：
- 輸入驗證是否完善？
- 是否有注入攻擊風險？
- 認證和授權是否正確？
- 敏感資料處理是否安全？

### 4. 效能審查

- 是否有 N+1 查詢問題？
- 是否有不必要的記憶體分配？
- 是否有適當的快取策略？
- 非同步操作是否正確處理？

### 5. 測試審查

- 測試覆蓋是否完整？
- 測試是否獨立且可重複？
- 是否涵蓋邊界情境？
- 測試命名是否清晰？

### 6. 產出審查報告

使用以下格式產出報告：

```markdown
# Review Report: SPEC-{number}

## Summary
- Spec compliance: X/Y criteria met
- Critical issues: N
- Warnings: N
- Suggestions: N

## Issues Found
### 🔴 Critical (must fix)
### 🟡 Warning (should fix)
### 🟢 Suggestion (nice to have)
```
