# 工作流：Bug 修復

修復生產環境或開發中發現的 Bug。

## 適用情境

- 生產環境的 Bug
- QA 發現的問題
- 使用者回報的問題
- 既有測試的失敗

## 精簡流程

```
分析 Bug → 撰寫精簡規格 → 修復 → 補充測試 → 審查
```

Bug 修復不需要完整的 7 階段流程，使用精簡版即可。

## 步驟詳解

### Step 1: 分析 Bug

確認以下資訊：
- **重現步驟**：如何穩定重現這個 Bug？
- **預期行為**：正確的行為是什麼？
- **實際行為**：目前的錯誤行為是什麼？
- **影響範圍**：影響多少使用者/功能？
- **根本原因**：問題的根本原因是什麼？

### Step 2: 撰寫精簡規格

不需要完整的功能規格書，但需要記錄：

```markdown
# Bug 修復：[Bug 標題]

> **編號**：BUG-{編號}
> **嚴重程度**：Critical / High / Medium / Low
> **關聯規格**：SPEC-{編號}（如有）

## 問題描述
[Bug 的描述]

## 重現步驟
1.
2.
3.

## 預期行為
[正確的行為]

## 實際行為
[錯誤的行為]

## 根本原因分析
[問題的根本原因]

## 修復方案
[計劃如何修復]

## 驗收條件
- [ ] Bug 不再重現
- [ ] 既有測試全部通過
- [ ] 新增的回歸測試通過
```

存放至 `docs/sdd/specs/BUG-{編號}-{描述}.md`

### Step 3: 修復

```
指令：/04-implement（引用 BUG-{編號}）
Agent：@sdd-developer
```

修復原則：
- 只修復 Bug，不做額外的「順便改善」
- 如果發現相關的其他問題，另開 Bug
- 確認修復不會造成回歸

### Step 4: 補充測試

```
指令：/05-write-tests（引用 BUG-{編號}）
Agent：@sdd-developer
```

- 寫一個可以重現原始 Bug 的測試（回歸測試）
- 確認此測試在修復前會失敗，修復後會通過

### Step 5: 快速審查

```
指令：/06-review（引用 BUG-{編號}）
Agent：@sdd-reviewer
```

審查重點：
- 修復是否正確解決根本原因？
- 是否有回歸風險？
- 回歸測試是否足夠？

## 完成條件

- [ ] Bug 不再重現
- [ ] 回歸測試已新增且通過
- [ ] 既有測試全部通過
- [ ] 審查通過
