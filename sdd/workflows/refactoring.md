# 工作流：重構

改善既有程式碼的結構、可讀性或效能，不改變外部行為。

## 適用情境

- 程式碼技術債清理
- 效能優化
- 架構模式調整
- 程式碼可讀性改善
- 依賴套件升級

## 流程

```
識別問題 → 規格化 → 設計 → 實作 → 測試驗證 → 審查
```

## 步驟詳解

### Step 1: 識別重構目標

明確列出要重構的項目和原因：

| 項目 | 目前的問題 | 期望狀態 |
|------|----------|---------|
|      |          |         |

### Step 2: 撰寫重構規格

```markdown
# 重構規格：[重構標題]

> **編號**：REFACTOR-{編號}
> **影響範圍**：[受影響的模組/檔案]

## 重構目標
[要解決什麼問題？]

## 約束條件
- 外部行為不變（所有既有測試必須通過）
- [其他約束]

## 重構項目
1. [具體的變更項目]
2. [具體的變更項目]

## 驗收條件
- [ ] 所有既有測試通過
- [ ] [具體的改善指標，如：函式行數 < 50]
- [ ] [具體的改善指標，如：回應時間 < 200ms]

## 不在範圍
- [明確不做的事]
```

### Step 3: 設計（如果是架構級重構）

對於架構級的重構，使用 `/03-design-system` 進行設計，並記錄 ADR。

小型重構可跳過此步驟。

### Step 4: 實作重構

```
指令：/04-implement（引用 REFACTOR-{編號}）
Agent：@sdd-developer
```

重構原則：
- **小步前進**：一次只做一個小的改動
- **持續綠燈**：每一步之後測試都要通過
- **不改行為**：重構不改變外部行為
- **不混合**：重構和功能變更不在同一個 commit

### Step 5: 驗證

- 運行所有既有測試，確認全部通過
- 對比重構前後的效能指標（如適用）
- 確認沒有引入新的依賴或複雜度

### Step 6: 審查

```
指令：/06-review（引用 REFACTOR-{編號}）
Agent：@sdd-reviewer
```

審查重點：
- 外部行為是否保持不變？
- 重構是否達到預期目標？
- 是否引入了新的問題？

## 完成條件

- [ ] 所有既有測試通過
- [ ] 重構達到規格中的改善指標
- [ ] 沒有改變外部行為
- [ ] 審查通過
