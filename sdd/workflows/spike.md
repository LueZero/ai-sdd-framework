# 工作流：技術探索（Spike）

研究性質的工作，目的是降低不確定性，為後續開發提供資訊。

## 適用情境

- 評估新技術或框架
- 研究可行性
- 建立概念驗證（PoC）
- 效能基準測試
- 第三方整合調研

## 流程

```
定義問題 → 研究 → 原型 → 結論
```

Spike 不走完整的 SDD 流程，重點在產出知識而非產品程式碼。

## 步驟詳解

### Step 1: 定義 Spike

```markdown
# 技術探索：[探索主題]

> **編號**：SPIKE-{編號}
> **時間限制**：[例如：2天]

## 研究問題
[我們想回答什麼問題？]

## 背景
[為什麼需要這個探索？]

## 成功標準
[什麼樣的結果算是探索成功？]
- [ ] [具體問題 1 有答案]
- [ ] [具體問題 2 有答案]
- [ ] [技術可行性已驗證]

## 研究範圍
- 要調研的技術/方案：
- 評估標準：
- 不在範圍：
```

### Step 2: 研究

使用 `@sdd-architect` 或 `@sdd-pm` 協助研究：
- 技術文件閱讀
- 替代方案比較
- 社群經驗蒐集

### Step 3: 建立原型（如需要）

```
指令：/04-implement
Agent：@sdd-developer
```

原型原則：
- **快速驗證**：不追求完美，追求速度
- **可丟棄**：原型程式碼不進入產品
- **聚焦問題**：只驗證特定的技術問題
- **記錄發現**：邊做邊記錄發現和問題

### Step 4: 產出結論

```markdown
# Spike 結論：[探索主題]

## 研究結果

### 問題 1: [問題]
**結論**：[回答]
**依據**：[證據/數據]

### 問題 2: [問題]
**結論**：[回答]
**依據**：[證據/數據]

## 方案比較

| 方案 | 優點 | 缺點 | 風險 | 建議 |
|------|------|------|------|------|
| A    |      |      |      |      |
| B    |      |      |      |      |

## 建議

[基於研究，我們建議......]

## 後續行動

- [ ] 建立 ADR 記錄決策
- [ ] 撰寫功能規格書（如決定開發）
- [ ] [其他行動]

## 原型程式碼

位置：[原型程式碼位置，或註明已刪除]
注意：原型程式碼不可直接用於產品
```

儲存至 `docs/sdd/decisions/SPIKE-{編號}-{主題}.md`

## 完成條件

- [ ] 所有研究問題都有結論
- [ ] 產出了 Spike 結論文件
- [ ] 明確的後續行動建議
- [ ] 如需要，已建立 ADR
