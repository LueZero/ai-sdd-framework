# Phase 4: 實作

依據規格書和設計文件，實作程式碼。

## 前置條件
- 功能規格書已完成（`docs/sdd/specs/`）
- 系統設計已完成（`docs/sdd/designs/`）
- 技術決策已記錄（`docs/sdd/decisions/`）

## 步驟

### 1. 閱讀文件
- 完整閱讀功能規格書
- 完整閱讀系統設計文件
- 閱讀相關的 ADR
- 了解專案上下文

### 2. 實作計劃

在開始編碼前，列出實作順序：

```
實作順序（由內到外）：
1. 資料模型 / Entity
2. Repository / Data Access Layer
3. Service / Business Logic
4. Controller / API Layer
5. UI 元件（如適用）
6. 整合配置
```

### 3. 編碼

依照計劃逐步實作，遵循：
- **規格一致性**：嚴格依照規格實作，不加入規格外功能
- **設計一致性**：遵循設計文件的元件結構和介面定義
- **專案慣例**：遵循既有的程式碼風格和模式
- **安全性**：遵循安全準則

### 4. 自我檢查

每個檔案完成後：
- [ ] 是否符合規格中的相關驗收條件？
- [ ] 是否遵循設計文件的介面定義？
- [ ] 是否遵循專案的命名和程式碼風格？
- [ ] 是否有適當的錯誤處理？
- [ ] 是否避免了硬編碼的敏感資訊？

## 實作原則

- **不過度工程**：只實作規格要求的功能
- **不擅自優化**：效能優化需有規格支持
- **保持一致**：與既有程式碼風格保持一致
- **可讀性優先**：清晰的程式碼優於聰明的技巧
- **漸進式**：一次專注一個元件，完成後再進入下一個
