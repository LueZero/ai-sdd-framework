# Phase 8: 規格變更請求

處理開發過程中的規格變更，確保變更以正確的順序執行（文件先於程式碼）。

## 前置條件
- 識別了需要變更的規格項目
- 知道變更的原因

## 步驟

### 1. 停止目前實作

如果正在實作中，先暫停，不要繼續實作不確定的部分。

### 2. 分類變更類型

判斷這是哪種類型的變更：

| 類型 | 描述 | 範例 |
|------|------|------|
| **澄清** | 規格含糊，需補充說明 | 驗收條件不夠具體 |
| **修正** | 規格有錯誤 | 欄位型別定義有誤 |
| **擴展** | 增加新需求 | 新增驗收條件 |
| **縮減** | 移除或簡化需求 | 縮小功能範圍 |
| **重定向** | 業務方向改變 | 整個功能重新設計 |

### 3. 撰寫變更請求

使用 `sdd/templates/change-request.tmpl.md` 範本，建立：
`docs/sdd/changes/CR-{編號}-{描述}.md`

### 4. 影響評估

識別受影響的所有工件：
- 哪些規格條目需要修改？
- 設計文件是否受影響？
- 已有程式碼是否需要重寫？
- 已有測試是否需要更新？
- 是否需要新增 ADR？

### 5. 按順序更新文件

**先更新文件，再修改程式碼：**

```
1. 更新規格書（在規格書末尾加入「變更歷史」記錄）
2. 更新設計文件（如果設計受影響）
3. 新增 ADR（如果有新的架構決策）
4. 修改程式碼（使用 /04-implement 重引用更新後的規格）
5. 更新測試（使用 /05-write-tests 重引用更新後的規格）
6. 重新審查（使用 /06-review 引用 CR 文件）
```

## 輕量決策指引

- **澄清/修正且不影響程式碼** → 只更新規格書
- **影響少量程式碼** → 完整 CR 文件 + 更新相關檔案
- **影響大範圍程式碼** → 完整 CR 文件 + 重跑受影響的 Phase
- **業務方向大改** → 評估是否需要重啟規格（回到 Phase 2）
